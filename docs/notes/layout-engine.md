# レイアウトエンジン — 検討メモと未解決事項

このノートはレイアウトエンジンに関するオーナーの検討記録の live note。
M1 Phase 3 で確定した決定は ADR (`docs/decisions/phase-3-layout-engine.md`)
を参照。本ノートには **ADR に書かれなかった代替案の検討記録** と、
**M2 以降の Open Questions** を残す。

---

## 1. M1 確定事項（参考）

Phase 3 で実装・合意された内容は ADR `phase-3-layout-engine.md` の DD-P3-001
以降を参照。要点のみ抜粋：

- 2-Pass (Measure / Arrange) を採用（Taffy 等の外部クレートは不採用）
- VStack / HStack / Rectangle のみ対応
- メモリモデルはエンジン所有 + 不透明ハンドル
- エラーは「API 呼び出しの正当性 (Strict)」と「レイアウト計算の完遂
  (Resilient)」を分離

これらが **採用された決定**。以下は採用されなかった代替案の記録。

---

## 2. 検討したが採用されなかった代替案（M1 時点）

将来 M2 以降でレイアウト方針を見直す際の出発点として保存。

### A. Snapshot 方式
キャッシュを Blob としてホスト側が管理・再注入する完全ステートレス化。
バインディング側でのキャッシュライフサイクル制御が複雑になるため M1 では
不採用。並列計算の余地を作りたくなったときに再検討する価値あり。

### B. Immediate Command 方式
永続ツリーを持たず、毎フレーム全命令を送信する方式。同期問題を回避できる
が、`.ui` DSL の宣言的モデルとの相性が悪く M1 では不採用。

### C. データ駆動型（中間表現方式）
UI DSL から「中間表現（バイナリ IR）」を生成し、独自レイアウト用の制約式や
計算ロジックをすべてデータとしてエンジンに流し込む案。ホスト言語への
コールバックを排除できる利点があるが、IR の表現力（条件分岐への対応）と
評価エンジン実装コストの不確実性が高く M1 スコープ外と判断。
**M2 で wasamoc がコード生成に進む際、もう一度検討の俎上に載せる価値あり。**

## 3. M2 以降の Open Questions

### 3.1 DPI スケーリングの局所化
物理ピクセルをエンジンが意識すべきか。DirectWrite のヒンティング精度に
影響する。M1 では論理ピクセルのみで動作。Grid / ScrollView 導入時に
再考が必要。

### 3.2 アクセシビリティ (AccessKit) との同期
レイアウト結果を UIA に反映するタイミングと負荷。M2 で AccessKit 統合が
入るときに設計判断が必要。候補：
- レイアウト確定時に UIA tree を一括更新
- ノード単位に lazy 反映（プロパティ変更ベース）

### 3.3 非同期 Measure の許容
画像読み込み待ちなど、サイズが確定しない要素の扱い。M1 のウィジェットには
該当なしだが、Image / Picture ウィジェットを M3 で導入するなら設計が必要。

### 3.4 キャッシュ無効化の粒度
現状（DD-P8-002）は size-affecting プロパティ変更でウィンドウ全体を
dirty 化。Grid / ScrollView の大規模化で再計算コストが問題化したら、
サブツリー単位の dirty 伝播へ移行する。指標：1,000 ノード規模での
再レイアウト時間。

### 3.5 拡張性 — 独自 Layout 関数の表現
DSL 側で `component MyLayout { ... }` のような形でユーザー定義レイアウトを
許すか。許す場合、ロジックの実行場所（host 言語 vs エンジン内 IR 評価）が
2 章 C 案と直結する。

---

## 4. 計測したい項目（実施未了）

M2/M3 で性能議論が必要になったら計測する：

- C ABI 性能試験: 1,000 ノード規模のツリー構築・更新コスト（言語境界往復
  の実コスト測定）
- キャッシュ有効性の測定: 局所変更時の DirectWrite オブジェクト再利用率
- バインディング体験の評価: 各ホスト言語での不透明ハンドルラッパーの
  自然さ・安全性
